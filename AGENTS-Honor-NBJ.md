## 关键规则

### 1. 文件读取
- 本知识库包含 `.txt`、`.md`、`.docx`、`.pdf` 四种格式，**全部直接读取**
- 每个模块目录下有 `INDEX.md` 索引文件，优先读取
- 深度分析时读取路由表指向的**所有相关文件**

### 2. 数据准确性（第一原则）
- **绝对禁止**凭记忆引用任何财务数据
- 所有财务数据必须通过 Python 代码实时获取（`yfinance`/`akshare`）
- 关键数据至少**双源交叉验证**
- 所有数据标注获取日期和来源
- 获取失败必须明确告知，**不得编造**

### 3. 分析流程（v3.0 八步升级版）

#### 第零步：预检飞行清单
开始任何分析前，快速完成8项预检：
1. **能力圈检查** — 该行业/公司是否在可理解范围内？
2. **数据可得性** — 是否有公开财务数据？
3. **生命周期判断** — 初创/成长/成熟/衰退期？→ 决定框架权重
4. **周期位置判断** — 行业周期和宏观周期的位置？
5. **市场情绪快照** — 市场共识是什么？过度乐观/悲观？
6. **信息质量评估** — 一手vs二手？关键信息盲区？
7. **既往分析回顾** — 检查`10-研究报告输出/`有无历史报告
8. **时间框架对齐** — 用户关心短期/中期/长期？

#### 第一步至第八步（详见copilot-instructions.md）
1. **需求识别 & 框架匹配** — 使用决策引擎（5种情景 × 标的属性 × 深度校准）
2. **数据先行** — Python获取 + 双源验证 + 数据快照表
3. **多框架并行分析** — 价值×成长×宏观三维，每框架含二阶思考
4. **前瞻性分析** — 领先指标映射 + 催化剂时间表 + 情景分析
5. **框架冲突仲裁** — 量化分歧 + 权重分配 + 分歧信号解读
6. **Kill-Switch检查** — 7项红线逐项检查
7. **概率化综合判断** — 贝叶斯加权 + TSR分解 + 盈亏平衡g*
8. **结构化输出** — 报告含领先指标监控表 + 二阶效应备注

### 4. 二阶思考要求（v3.0 新增）

> **每个分析结论必须附带二阶推演。**

在得出任何判断后，必须追问：
- **"然后呢？"** — 如果A发生，B会跟着发生吗？识别连锁反应
- **"共识在哪？"** — 市场共识是什么？我的独立判断哪里不同？
- **"线性还是非线性？"** — 这个变化是渐进的还是有拐点/指数级的？
- **"反身性？"** — 市场预期本身是否会改变现实？

### 5. 范式归类要求（v3.0 新增）

> **每个标的必须完成范式归类和陷阱检查。**

#### 五类正范式（必选其一）
| 范式 | 核心特征 | 关键指标 |
|------|----------|----------|
| 质量复利 | 高ROIC+可持续增长+强护城河 | ROIC>15%, g>8%, MROIC稳定或上升 |
| 平台生态 | 网络效应+飞轮+多边平台 | 用户增长+变现率+留存率 |
| GARP | 合理估值+稳健成长 | PEG<1.5, ROE>12%, g>12% |
| 低估催化 | 深度折价+可识别催化剂 | P/EPV<0.8, 催化剂明确 |
| 现金回报防御 | 高股息+低估值+稳定FCF | 股息率>3%, FCF收益率>6% |

#### 六类负范式陷阱（必检）
| 陷阱 | 检测方法 | Kill-Switch阈值 |
|------|----------|-----------------|
| 倍数幻觉 | 正常化盈利 vs 报告盈利 | 差异>30% |
| 杠杆驱动 | DuPont分解ROE来源 | 财务杠杆贡献>50% |
| 伪成长 | 有机增长率 vs 总增长率 | 有机增长<总增长50% |
| 深度价值陷阱 | 资产减值风险评估 | 重置成本>账面价值30% |
| 并表型增长 | 少数股东利润占比 | 少数股东利润>30% |
| 单一依赖 | Top客户/产品集中度 | 单一来源>50% |

### 6. Kill-Switch 机制（v3.0 新增）

以下任一条件触发→**立即停止推荐，降为"观察"：**

| 触发条件 | 严重程度 |
|---|---|
| 核心财务数据无法获取或不可靠 | 致命 |
| 商业模式无法在30秒内向外行解释 | 致命 |
| 管理层信誉问题/大额关联交易/频繁变更审计师 | 致命 |
| 有息负债/EBITDA > 4x 且无明确去杠杆路径 | 严重 |
| 当前价格 > EPV（无护城河企业），或安全边际 < 20% | 严重 |
| 收入>50%来自单一客户/产品/地区 且无改善 | 警告 |
| 所有框架+市场共识一致看好，无独立信息优势 | 警告 |

### 9. 债券市场数据获取（可交易级别）

**债券策略报告必须先获取硬数据，避免"宏大叙事+空洞结论"**

```bash
# 获取完整债券市场快照
python bond_data.py --all

# 分项获取
python bond_data.py --curve   # 收益率曲线 + Carry/Rolldown
python bond_data.py --macro   # CPI/PPI/货币供应
python bond_data.py --policy  # LPR政策利率
```

**核心验真指标（3个硬指标）**：
1. **收益率曲线**：10Y国债、期限利差（10Y-2Y）、Carry+Rolldown
2. **通胀数据**：CPI同比、核心CPI
3. **政策利率**：LPR 1Y/5Y、M2-M1剪刀差

**债券策略报告流程**：
1. 先运行 `python bond_data.py --all` 获取实时数据
2. 基于数据确定久期策略（而非先写结论再找数据）
3. 用 Carry+Rolldown 计算各期限"单位DV01收益"
4. 风控阈值与持仓风险映射（DV01、VAR）

## Workflow Orchestration 工作流编排

### 12. 规划优先（Plan Node Default）
- 任何非平凡任务（3步以上或涉及架构决策）→ **先进入规划模式**
- 如果执行中出现偏差，**立即停下来重新规划**，不要硬推
- 验证步骤也需要规划，不只是构建阶段
- 开工前撰写详细规格说明，减少歧义

### 13. 子代理策略（Subagent Strategy）
- 大方使用子代理，保持主上下文窗口干净
- 研究、探索、并行分析等任务 → 分流到子代理
- 复杂问题上用更多算力（多子代理并行）
- 每个子代理只做一件事，专注执行

### 14. 自我改进循环（Self-Improvement Loop）
- 每次收到用户纠正 → 立即将模式记录到 `tasks/lessons.md`
- 为自己编写规则，防止同类错误再次发生
- 持续迭代这些经验教训，直到错误率下降
- 每次会话开始时回顾相关项目的经验

### 15. 完成前验证（Verification Before Done）
- **未经证明有效，不得标记任务完成**
- 对比变更前后的行为差异（如有关联）
- 自问：「一位 Staff Engineer 会批准这个吗？」
- 运行测试、检查日志、展示正确性

### 16. 追求优雅（Demand Elegance — Balanced）
- 非平凡修改：暂停并问「有没有更优雅的方式？」
- 如果修复方案感觉是 hack：「基于已知的全部信息，实现优雅方案」
- 简单明显的修复不必过度工程化
- 呈现前先挑战自己的方案

### 17. 自主修复（Autonomous Bug Fixing）
- 收到 bug 报告后直接修复，不要等用户手把手指导
- 对着日志、报错、失败的测试逐一排查并解决
- 用户无需为 bug 做上下文切换
- 主动修复失败的 CI 测试，无需被告知怎么做

## Task Management 任务管理

1. **Plan First** — 将计划写入 `tasks/todo.md`，列出可勾选项
2. **Verify Plan** — 开始实现前先确认计划
3. **Track Progress** — 逐项标记完成
4. **Explain Changes** — 每步给出高层摘要
5. **Document Results** — 在 `tasks/todo.md` 添加回顾小节
6. **Capture Lessons** — 收到纠正后更新 `tasks/lessons.md`

## Core Principles 核心执行原则

- **简洁优先（Simplicity First）**：每次改动尽可能简单，影响最少的代码
- **不偷懒（No Laziness）**：找到根因，不打临时补丁，保持 Senior Developer 标准
- **最小影响（Minimal Impact）**：只动必须动的地方，避免引入新 bug
