# Superpowers × 投研框架 集成指南

> 日期：2026-02-26  
> 学习来源：https://github.com/obra/superpowers（v4.3.1+）

---

## 核心思想迁移

Superpowers 是为**软件开发**设计的，但它的本质是一套**结构化工作流哲学**，可以完整迁移到投研分析中：

| Superpowers 概念 | 投研框架对应 | 铁律 |
|---|---|---|
| **TDD：先写测试** | 先写投资论点（可证伪条件），再收集数据 | 不得凭感觉先下结论 |
| **Brainstorming：明确需求** | 研究前先明确：研究问题/时间框架/深度等级 | L3+ 必须先确认框架组合 |
| **Writing Plans：拆解任务** | 把分析拆成子任务（数据获取→估值→成长→宏观→综合） | 超过3步先写 todo.md |
| **Systematic Debugging：追根溯源** | 框架结论矛盾时不靠直觉，系统排查根本原因 | 不得在未明白原因前随意调整结论 |
| **Verification Before Completion** | 所有财务数据必须代码实时获取，不得凭记忆 | 声明任何结论前先运行代码验证 |
| **Kill-Switch 机制** | 7项红线任一触发→立即降为"观察" | 等同于 Superpowers 的 HARD-GATE |

## 投研版 Superpowers 工作流

### 第零步：`brainstorming`（研究问题澄清）

**触发条件：** 用户提出任何分析请求

**执行检查清单：**
1. 研究标的是什么？（公司/行业/宏观/决策）
2. 对应哪种分析情景？（S1个股/S2行业/S3宏观/S4投资决策/S5十倍股）
3. 分析深度是 L1-L6？（快速扫描/中度研究/深度研报/极致研究/买方决策/沙盘推演）
4. 时间框架是多少？（短期<1年/中期1-3年/长期3年+）
5. 用户关心的核心问题是什么？（估值？护城河？拐点？风险？）

**HARD-GATE：在获得这5个问题的答案之前，不得开始任何分析。**

---

### 第一步：`writing-plans`（研究计划）

**触发条件：** 研究需求已澄清，深度 L3 及以上

**研究计划模板（保存到 `tasks/todo.md`）：**

```markdown
# [标的名] 研究计划 — [日期]

**研究问题：** [一句话]
**分析情景：** [S1-S5]
**分析深度：** [L1-L6]
**时间框架：** [短/中/长期]
**框架组合：** [选中的2-6个框架]
**数据来源：** yfinance / akshare / 双源交叉

---

### 任务1：数据获取（预计5分钟）
- 获取：PE/PB/PS/ROE/营收增速/FCF/股息率
- 工具：Python (yfinance + akshare)
- 验证：双源交叉，偏差>10%标注

### 任务2：估值分析（预计10分钟）
- 框架：格林沃尔德三重估值 / 市赚率PR / 散户乙十年回本
- 输出：安全边际评估 + 目标价区间

### 任务3：成长分析（预计10分钟）
- 框架：10X Alpha / 贝叶斯拐点 / BG韧性
- 输出：成长持续性评估 + 置信度

### 任务4：宏观/周期分析（预计5分钟）
- 框架：三重共振 / 周金涛康波
- 输出：时机评估

### 任务5：框架综合 & Kill-Switch检查（预计5分钟）
- 执行框架冲突仲裁
- 逐项检查7项Kill-Switch红线
- Q-G-P-R百分制打分

### 任务6：生成报告（预计5分钟）
- 模板：`10-研究报告输出/`
- 包含：执行摘要/数据快照/多框架矩阵/情景分析/可证伪条件
```

---

### 第二步：投研版 `test-driven-development`

**核心迁移：先写"测试"（可证伪的投资论点），再收集数据。**

```
传统错误顺序（BAD）：
  收集数据 → 看到数据好 → 写出看多论点 → 找数据支持

Superpowers 投研顺序（GOOD）：
  1. 先写投资论点："我认为X公司具有护城河，因为Y。如果发生Z，说明我错了。"
  2. 再运行代码获取数据
  3. 用数据验证/推翻论点
  4. 贝叶斯更新置信度
```

**投研 TDD 铁律：**
- 每个投资论点必须有可证伪条件（"如果X发生，说明判断错误"）
- 不得凭印象说"ROE约15%"——必须运行代码获取实时数据
- 数据与论点矛盾时：**不得修改论点使其符合数据**，应重新分析

---

### 第三步：投研版 `systematic-debugging`

**触发条件：** 多框架结论矛盾，或数据质量存疑

**4阶段调试：**

#### Phase 1：追根溯源
- 框架A说便宜，框架B说贵——哪个框架的假设有问题？
- 数据来源可靠吗？是否存在一次性项目影响当期利润？
- 近期有什么变化导致两个框架结论分歧？

#### Phase 2：模式分析
- 同行业可比公司的估值是多少？
- 历史上该公司在类似情景下怎么表现？

#### Phase 3：单一假设检验
- 形成一个假设：「我认为分歧的原因是X」
- 最小化测试：调整一个变量，看矛盾是否消失

#### Phase 4：结论修正
- 用Q-G-P-R打分卡做最终量化综合
- 标注置信度区间

**铁律：不得在未明白矛盾根源时强行给出结论。说"暂无法判断"比编造一致性更诚实。**

---

### 第四步：投研版 `verification-before-completion`

**核心迁移：在任何声明数据/结论前，必须用代码验证。**

```
❌ BAD（违规声明）：
   "茅台当前 PE 约30倍。"
   "该公司ROE稳定在20%左右。"

✅ GOOD（验证后声明）：
   [运行 yfinance/akshare 代码]
   [获取：PE = 28.5x，数据日期：2026-02-26，来源：yfinance]
   "茅台当前 PE 为 28.5 倍（2026-02-26，yfinance）。"
```

**报告提交前必过5关：**
1. ✅ 所有财务数据已代码获取并注明日期/来源
2. ✅ 建立了完整逻辑链（论点→数据→结论→可证伪条件）
3. ✅ Kill-Switch 7项全部检查（无触发 or 标注已触发项）
4. ✅ 每个投资论点有可证伪条件
5. ✅ 执行摘要3句话内说清楚

## Kill-Switch = Superpowers HARD-GATE

以下任一触发→**立即停止推荐，降为"观察"**（等同于 Superpowers 的 HARD-GATE）：

| Kill-Switch | 触发条件 | 等同于 Superpowers |
|---|---|---|
| 数据缺失 | 核心财务数据无法获取 | 测试无法运行 |
| 能力圈越界 | 商业模式无法30秒解释 | 无法写出清晰需求 |
| 管理层红旗 | 信誉问题/关联交易/换审计师 | 代码分析前发现架构缺陷 |
| 杠杆红线 | 有息负债/EBITDA > 4x | 关键测试永远失败 |
| 安全边际不足 | 价格 > EPV（无护城河） | 验证无法通过 |
| 单一依赖 | 收入>50%单一来源 | 单点故障风险 |
| 全面共识 | 无独立信息优势 | 结论已是公开知识 |

## 快捷映射

| 投研指令 | 对应 Superpowers 技能 | 核心行为 |
|---|---|---|
| `@分析 [公司]` | brainstorming → writing-plans → executing-plans | 先澄清需求，再写计划，再执行 |
| `@估值 [公司]` | test-driven-development + verification | 先写论点，再获取数据验证 |
| `@拐点 [公司]` | systematic-debugging | 4阶段追根溯源找拐点信号 |
| `@L3/L4/L5/L6` | subagent-driven-development | 每个分析任务独立子流程+双阶段审查 |
| 声明任何数据 | verification-before-completion | 运行代码→读取输出→再声明 |
| 框架结论矛盾 | systematic-debugging | 不猜不绕，系统排查根本原因 |
| 新增分析框架 | writing-skills | 按 TDD 流程：RED（失败案例）→ GREEN（写技能）→ REFACTOR |

## 二阶思考（Superpowers 扩展维度）

Superpowers 的核心精神是"不要凭感觉，要走流程"。在投研场景中，额外加入：

1. **"然后会发生什么？"** — 每个投资论点都要做二阶推演
2. **"共识在哪里？"** — 先找市场共识，再找独立判断
3. **"线性还是非线性？"** — 识别拐点与反身性
4. **"我的论点可以被证伪吗？"** — 不可证伪 = 不是分析
