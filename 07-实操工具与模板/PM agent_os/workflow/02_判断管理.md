# Phase 2: 判断管理

## 触发条件
用户说 **"看看XX"**、**"分析XX"**、**"我觉得XX"**

## 核心原则

> 每一个判断都必须有证伪条件。
> 你必须提前说明，在什么情况下你会认为自己错了。
> 没有证伪条件的"判断"，只是一种希望。

## 建立新判断

### 必须收集的信息
当用户提出一个新的交易想法时，必须引导他明确以下要素：

| 要素 | 说明 | 是否必填 |
|------|------|---------|
| symbol | 标的代码 | 必填 |
| symbol_name | 标的名称 | 必填 |
| direction | 方向 (long/short/neutral) | 必填 |
| thesis | 核心判断逻辑 | 必填 |
| confidence | 置信度 (0-1) | 必填 |
| time_horizon | 时间框架 | 必填 |
| key_variables | 支撑判断的关键变量 | 必填 |
| invalidation_conditions | 证伪条件 | **必填！不可跳过！** |

### AI 质量评估
建立判断后，你必须对判断进行评估：
1. 逻辑是否自洽？
2. 证伪条件是否可操作（具体、可观测、有时间限制）？
3. 是否存在常见的认知偏差？
   - 确认偏差：只看支持自己的证据
   - 锚定效应：被某个价格锚定
   - 近因偏差：过度受近期事件影响
   - 从众心理：因为别人也在买就跟
4. 这个判断和已有判断是否矛盾？
5. 风险提示

### 输出格式
```
──────────────────────────────────────────────────

──────────────────────────────────────────────────
  标的:     600519 (贵州茅台)
  方向:     做多

  时间框架: 短期（1-2周）

    白酒消费复苏，春节数据超预期...
    
  关键变量:

```

## 判断状态管理

### 状态转换
```
  创建 → active（活跃）
           │
           ├─→ weakened（松动）—— 关键变量出现不利变化
           │      │
           │      ├─→ active（回到活跃）—— 松动迹象消除
           │      └─→ invalidated（证伪）—— 证伪条件触发
           │
           ├─→ invalidated（证伪）—— 证伪条件直接触发
           │
           └─→ expired（过期）—— 超出时间框架
```

### 松动迹象的判定
判断松动 ≠ 判断错误。松动意味着：
- 部分关键变量出现了不利变化
- 但尚未触及证伪条件
- 需要交易者加强关注，降低仓位

### 证伪的判定
判断被证伪是一件好事。它意味着你的系统在工作。
证伪后：
- 基于该判断的所有决策应被取消
- 相关持仓应考虑平仓
- 不得在判断被证伪后以同一逻辑重新开仓

## 状态文件更新
新增或变更判断后需要更新：
- `state/judgments.json` — 添加新判断或更新状态
