# 高管指令｜投资框架是否应 Skill 化（v1.0）

## 一、决策目标
在不牺牲研究质量与灵活性的前提下，提升 AI 执行稳定性、复用效率与维护效率。

## 二、决策问题
是否应将现有投资框架文档（大量 .txt/.md/.docx/.pdf）整体转换为 skill 文件？

## 三、最高原则（按优先级）
1. 研究正确性 > 执行速度
2. 数据先行与可审计性 > 文档形式统一
3. 可维护性 > 一次性重构
4. 渐进改造 > 全量迁移

## 四、判定标准（五维）
- 复用频率：同类流程每周是否高频重复
- 歧义程度：流程步骤是否稳定、低歧义
- 维护成本：更新一次是否会引发多处同步修改
- 召回准确：路由是否能稳定命中正确框架
- 风险敞口：误路由/过度模板化是否会降低分析质量

## 五、转换边界（硬规则）
### A. 必须 Skill 化（流程资产）
- 高复用、步骤稳定、可检查的工作流
- 例如：@分析、@估值、@护城河、@宏观、@债券、@日志

### B. 不应 Skill 化（知识资产）
- 以观点密度和案例细节为主、需要灵活抽取的框架原文
- 例如：大师思想原文、行业深度材料、案例库

### C. 条件 Skill 化（半结构资产）
- 具备明确输入输出但需依赖上下文判断的模块
- 采用“Skill 外壳 + 文档检索内核”模式

## 六、默认架构决策
采用“混合架构”而非“全量 Skill 化”：
- Skill 负责：路由、流程、检查清单、输出模板
- 文档库负责：框架知识本体、案例细节、补充证据

## 七、执行红线
- 禁止把全部框架原文机械改写为 skill（会导致维护爆炸与信息损失）
- 禁止出现双真源（同一规则在多处各自演化）

## 八、90天执行策略
1) 巩固现有 InvestmentCRO skill（流程层）
2) 补齐缺失 workflow（如 @打分/@情景/@陷阱）
3) 为 INDEX 建立轻量元数据（标签、适用场景、反向筛选）
4) 保持框架正文在原目录，按需检索，不做全量迁移

## 九、成功标准（KPI）
- 指令命中率提升（错误路由下降）
- 单次分析准备时间下降
- 报告结构一致性提升
- 框架更新时的维护触点减少

## 十、一句话裁决
“流程 Skill 化，知识文档化；先混合、后迭代，不做全量迁移。”
