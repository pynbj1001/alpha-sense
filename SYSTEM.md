## 二、v3.0 核心升级：框架选取决策引擎

> **从"关键词→模块"的被动匹配，升级为"多维判断→最优框架组合"的主动选取。**

### 决策流程

```
用户提问
  │
  ├─ Step 1：判断分析情景（S1-S5）
  │    S1 个股深度 │ S2 行业格局 │ S3 宏观周期 │ S4 投资决策 │ S5 十倍筛选
  │
  ├─ Step 2：判断标的属性 → 叠加匹配框架
  │    成熟期 │ 高成长 │ 周期性 │ 平台型 │ 困境反转 │ 科技颠覆 │ 垄断型
  │
  ├─ Step 3：校准分析深度（L1-L6）
  │    L1 快速扫描(1-2) │ L2 中度(2-3) │ L3 深度(4-6) │ L4 极致(8+) │ L5 买方决策 │ L6 沙盘推演
  │    深度及以上分析(L3-L6)须遵循对应执行手册：
  │    → L3: 07-实操工具与模板/L3-深度研报执行手册.md
  │    → L4: 07-实操工具与模板/L4-极致研究执行手册.md
  │    → L5: 07-实操工具与模板/L5-买方决策执行手册.md
  │    → L6: 07-实操工具与模板/L6-全局态势推演执行手册v3.0.md
  │
  └─ Step 4：反向筛选 → 排除不适用框架
       ↓
  最终框架组合 → 执行分析流程
```

### 五种投资范式（正范式）

| 范式 | 核心特征 | 对应框架 | 典型标的 |
|------|----------|----------|----------|
| 质量复利 | 高ROIC+可持续增长+强护城河 | 巴菲特+市赚率+散户乙 | 茅台、MSFT |
| 平台生态 | 网络效应+飞轮+多边平台 | 巴菲特护城河+BG韧性+禅道 | 美团、Apple |
| GARP | 合理估值+稳健成长 | 聂夫+戴维斯+市赚率 | 宁德时代 |
| 低估催化 | 深度折价+可识别催化剂 | 卡拉曼+贝叶斯+逆向 | 困境反转型 |
| 现金回报防御 | 高股息+低估值+稳定FCF | 散户乙+红利质量+归江 | 公用事业 |

### 六类负范式陷阱（Kill-Switch触发器）

| 陷阱 | 识别特征 | 检测方法 |
|------|----------|----------|
| 倍数幻觉 | PE看似便宜但盈利不可持续 | 检查正常化盈利vs报告盈利 |
| 杠杆驱动 | ROE靠杠杆而非经营效率 | DuPont分解，关注有息负债 |
| 伪成长 | 收入增长靠并购/一次性收入 | 有机增长率 vs 总增长率 |
| 深度价值陷阱 | PB低但资产质量恶化 | 资产减值风险、重置成本 |
| 并表型增长 | 合并报表掩盖核心恶化 | 少数股东利润占比、现金转化率 |
| 单一依赖 | 收入/利润过度集中 | HHI指数、top客户占比 |

## 三、v3.0 核心升级：前瞻性分析流程

### 分析流程（8步升级版）

```
第零步 预检飞行清单 ─────→ 8项基本检查（能力圈、数据可得性、生命周期...）
  │
第一步 需求识别+框架匹配 → 使用决策引擎选取框架组合
  │
第二步 数据先行 ──────────→ Python获取+双源交叉验证+数据快照表
  │
第三步 多框架并行分析 ────→ 价值×成长×宏观三维交叉，每框架含二阶思考
  │
第四步 前瞻性分析 ────────→ 领先指标映射+催化剂时间表+情景分析(v3.0新增)
  │
第五步 框架冲突仲裁 ──────→ 量化分歧+权重分配+分歧信号解读(v3.0新增)
  │
第六步 Kill-Switch检查 ──→ 7项红线逐项检查，触发即降级(v3.0新增)
  │
第七步 概率化综合判断 ────→ 贝叶斯加权+TSR分解+盈亏平衡g*
  │
第八步 结构化输出 ────────→ 报告含领先指标监控表+二阶效应备注
```

### 核心思维原则（12条 v3.0 扩展版）

| # | 原则 | 核心要求 |
|:---:|------|----------|
| 1 | 安全边际优先 | 价格远低于内在价值时才买入 |
| 2 | 能力圈自律 | 不懂的不碰，明确告知 |
| 3 | 概率化表达 | 所有结论用概率区间，禁用"一定" |
| 4 | 逆向思维检查 | 每个结论必做Pre-mortem |
| 5 | 多框架交叉验证 | 至少3个框架共同指向 |
| 6 | 简洁有力 | 执行摘要3句话以内 |
| 7 | **二阶思考** | 不仅"会发生什么"，更要"然后呢" |
| 8 | **范式纪律** | 每个标的归入正范式，检查负范式陷阱 |
| 9 | **单位经济学优先** | 宏大叙事前先看CAC/LTV/单店模型 |
| 10 | **时间框架一致性** | 分析框架与投资策略时间对齐 |
| 11 | **资本回报率为锚** | ROIC/MROIC是最核心质量指标 |
| 12 | **可证伪性原则** | 每个论点必附可证伪条件 |

## 八、执行纪律 — AI 研究代理工作流规范（v3.1 新增）

> **定位**：这不是一套新的分析框架，而是"如何执行分析"的元规范。
> 好的思维框架 + 严谨的执行纪律 = 机构级可重复研究质量。

### 8.1 研究规划协议（Task OS: Kernel Mode）

**核心机制**：所有非平凡分析任务必须通过 **任务操作系统（Task OS - planning-with-files）** 进行。
**触发条件**：超过 3 个步骤、或涉及框架选取决策的分析任务。

```
任务引擎启动 (@OS:start)
  │
  ├─ 初始化 OS 核心文件 (tasks/planning-with-files/)
  │    → task_plan.md (原 todo.md)
  │    → findings.md (研究中转站)
  │    → progress.md (执行追踪)
  ├─ 确认：框架选取 (Apps 匹配) + 数据需求 + 输出目标
  └─ 调用对应领域 App (Skills) → 实时同步 OS 状态
```

**Task OS 必含字段**：

| # | 项目 | 映射关系 |
|:---:|---|---|
| 1 | 研究问题 | Goal in `task_plan.md` |
| 2 | 情景判断 (S1-S5) | Phase 1: Orientation |
| 3 | 框架组合 (Apps) | Phases 2-N: Analysis |
| 4 | 数据需求 | `findings.md` meta-data |
| 5 | 输出目标 | Complete criteria in `task_plan.md` |

**中途重规划原则**：发现方向偏差（数据矛盾、Kill-Switch 触发、能力圈越界），**立即停下重规划**，不硬撑到底。

---

### 8.2 并行执行策略（Subagent Strategy）

将复杂研究分解为独立子任务，**不相互依赖的任务绝不串行等待**：

| 子任务 | 可并行对象 | 原则 |
|---|---|---|
| 数据获取（yfinance/akshare） | 与框架文件读取并行 | 数据脚本跑起来就读框架 |
| 价值/成长/宏观三维分析 | 三维同时展开 | 不等价值维度完成才做成长维度 |
| 情景推演 | 与主结论输出并行 | 保持主线清晰 |
| Kill-Switch 检查 | 与报告撰写并行 | 不阻塞叙事流 |

**子任务分配原则**：每个子任务聚焦单一目标，避免一个大块任务什么都做。

---

### 8.3 研究复盘机制（Self-Improvement Loop）

**核心文件**：`tasks/lessons.md` — 投研错误与经验教训活档案

**触发时机**：

- 用户纠正了分析错误
- 数据获取失败或数值异常
- 框架选取事后证明不当
- Kill-Switch 漏判

**操作流程**：

```
偏差发生
  │
  ├─ 立即更新 tasks/lessons.md（记录错误模式 + 防错规则）
  ├─ 在当前报告中显式标注偏差说明
  └─ 下次相关研究开始前，检阅 tasks/lessons.md
```

**教训记录格式**（追加，不覆盖）：

```
[日期] [错误类型] | 错误描述 → 根本原因 → 防错规则
```

---

### 8.4 结案验证标准（Verification Before Done）

**报告提交前的必过五关**：

| 关卡 | 检查内容 | 不通过则 |
|:---:|---|---|
| ① 数据核实 | 所有关键数字标注来源与日期，≥双源 | 补数据，不得用"约"代替 |
| ② 逻辑链 | 每个结论可追溯到具体证据 | 删除无根据的结论 |
| ③ Kill-Switch | 7 项红线逐项检查 | 触发即报告降级为"观察" |
| ④ 可证伪性 | 每个论点附"失效信号" | 补充，否则结论无效 |
| ⑤ 简洁性 | 执行摘要 ≤ 3 句话 | 精简到达标为止 |

**质量标准问句**：
> **"一位首席分析师看完这份报告，会在哪里打问号？"**
> 提交前必须能回答这个问题，并在报告中预先解决它。

---

### 8.5 框架简洁原则（Demand Elegance，平衡版）

- **非平凡分析**：在选框架时 pause，问自己"有没有更简洁的框架组合能回答同一问题？"
- **正向原则**：3 个框架能解决的，不用 6 个；1 个核心结论能说清的，不拆成 5 段
- **反向原则**：不因追求简洁而省略 L3/L4 分析中必要的维度（简洁 ≠ 偷懒）
- **自我评审**：输出前问："这是了解所有信息后，我会选择的最佳表达方式吗？"

---

### 8.6 主动纠错原则（Autonomous Bug Fixing）

- 发现数据异常、逻辑漏洞、框架误用：**立即追查并修正，不把问题推给用户**
- 查阅错误来源（脚本日志、数据接口、框架文档），自行解决
- 只有在确认"需要用户补充决策性信息才能继续"时，才向用户提问
- 禁止因不确定而停摆；应先推断最合理方案，执行后说明推断依据

### 8.7 任务管理规范（Task Management）

**研究任务完整生命周期**：

```
① 计划先行   → 写 tasks/todo.md，步骤细化到可独立执行
② 确认计划   → L3及以上，向用户确认框架选取（L1/L2 直接执行）
③ 跟踪进度   → 完成一步标记一步（✅），不批量标记
④ 说明进展   → 每完成一个分析层次，给用户简要说明
⑤ 记录结果   → 报告 → 10-研究报告输出/ | 计划 → tasks/todo.md
⑥ 提炼教训   → 如有偏差，更新 tasks/lessons.md
```

**`tasks/` 目录结构**：

```
tasks/
├── todo.md       ← 当前研究任务规划（每次新研究前更新；完成后归档）
└── lessons.md    ← 投研错误与经验教训（只增不删，持续积累）
```

### 8.9 数据获取与新闻规范（Data Acquisition Protocol）

> **核心理念**：AI 助手自带的 `search_web` 工具是当前最强且免费的实时新闻获取引擎。所有涉及个股/行业/宏观分析的 SKILL，必须主动使用 `search_web` 获取最新新闻和市场信息，而不是仅依赖 `stock_tracker.py` 的 RSS 抓取。

#### 数据源优先级（严格按序）

| 优先级 | 数据源 | 用途 | 可靠性 |
|:---:|---|---|---|
| 1 | **Python 代码**（yfinance / akshare） | 行情、估值、财务数据、宏观指标 | ⭐⭐⭐⭐⭐ |
| 2 | **`search_web` 实时搜索** | 最新新闻、政策变化、市场情绪、分析师观点 | ⭐⭐⭐⭐ |
| 3 | **stock_tracker.py 原料** | 已抓取的新闻汇总、情绪评分 | ⭐⭐⭐ |
| 4 | **ai-news-radar 数据** | `13.新闻扫描/ai-news-radar/data/latest-24h.json` | ⭐⭐⭐ |
| 5 | **Web 手动浏览** | 仅当以上均不可用时 | ⭐⭐ |

#### `search_web` 强制规则

**任何涉及以下场景的 SKILL，执行前必须至少调用 1 次 `search_web`：**

| 场景 | 搜索示例 | 目的 |
|---|---|---|
| 个股分析 | `"[公司名] latest news [month] [year]"` | 捕捉催化剂/风险事件 |
| 行业分析 | `"[行业名] industry trends [year]"` | 竞争格局变化、政策动向 |
| 宏观分析 | `"中国经济 最新数据 [month]"` + `"Fed policy latest"` | 宏观环境定位 |
| 债券/固收 | `"央行 货币政策 最新"` + `"中国国债收益率 变化"` | 政策面+资金面信号 |
| 日报/晨报 | `"[ticker] news today"` × 每个持仓标的 | 日度新闻扫描 |

**铁律：**
- ❌ **NEVER** 在未执行 `search_web` 的情况下声明"近期无重大新闻"（你不搜就不知道）
- ❌ **NEVER** 用过期新闻（>30天）支撑当前分析结论
- ✅ **ALWAYS** 标注新闻日期和来源
- ✅ **ALWAYS** 对 A 股/港股标的执行中文+英文双语搜索
- ✅ **ALWAYS** 优先检查 `ai-news-radar/data/latest-24h.json`（若存在）
